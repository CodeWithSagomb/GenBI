FILTRES TEMPORELS :
• "par mois"            → GROUP BY sale_month ORDER BY sale_month  (fct_sales, sans WHERE)
• "ce mois"             → WHERE sale_month = EXTRACT(MONTH FROM CURRENT_DATE)::int AND sale_year = EXTRACT(YEAR FROM CURRENT_DATE)::int
• "mars 2026" / mois précis → WHERE sale_month = 3 AND sale_year = 2026
• fct_missed_sales      → missed_month / missed_year  (pas sale_month / sale_year)
• fct_purchases         → order_date pour filtres temporels (pas sale_date)
• stg_raw__sale_details → pas de sale_date — filtrer via JOIN sur marts.fct_sales.sale_id
• dim_clients (par mois) → GROUP BY DATE_TRUNC('month', created_at) ORDER BY DATE_TRUNC('month', created_at)  — NE PAS utiliser EXTRACT(MONTH) seul (fusionne les années)

RÈGLES TECHNIQUES :
1. Préfixe schéma obligatoire  →  marts.fct_sales ✓   fct_sales ✗
2. Alias standards : fct_sales=s · dim_products=pd · dim_stocks=sk · fct_purchases=fp · dim_clients=c · dim_insurers=ins · fct_wholesaler_returns=wr
3. RLS actif : ne pas ajouter WHERE pharmacy_id (filtrage automatique)
4. Colonnes : utiliser uniquement celles du schéma ci-dessous — ne rien inventer
5. Marge : utiliser marts.fct_purchases (a product_id) — fct_sales N'A PAS product_id
6. Produit par vente : passer par staging.stg_raw__sale_details sd JOIN marts.fct_sales s ON sd.sale_id = s.sale_id
7. COUNT DISTINCT produits → toujours via sd.product_id qualifié — fct_sales n'a pas product_id
   ✓  SELECT COUNT(DISTINCT sd.product_id) FROM marts.fct_sales s JOIN staging.stg_raw__sale_details sd ON sd.sale_id = s.sale_id
   ✗  SELECT COUNT(DISTINCT product_id) ...
8. Différence de dates en jours → (date_col - CURRENT_DATE) retourne un INTEGER directement — ne jamais utiliser INTERVAL pour calculer des jours
   Lots expirant dans X jours → toujours écrire la condition complète avec BETWEEN
   ✓  WHERE (sk.expiration_date - CURRENT_DATE) BETWEEN 0 AND 30
   ✓  WHERE (sk.expiration_date - CURRENT_DATE) < 30 AND (sk.expiration_date - CURRENT_DATE) >= 0
   ✗  WHERE sk.days_until_expiry  (incomplet — toujours inclure l'opérateur et la valeur)
   ✗  WHERE (sk.expiration_date - CURRENT_DATE) * INTERVAL '1 day' ...
9. dim_stocks JOIN ventes → toujours GROUP BY pd.commercial_name + MIN(sk.quantity_in_stock) — un produit a plusieurs lots, le JOIN crée des doublons sinon
   ✓  SELECT pd.commercial_name, MIN(sk.quantity_in_stock) AS stock ... GROUP BY pd.commercial_name
   ✗  SELECT pd.commercial_name, sk.quantity_in_stock ... (sans GROUP BY → lignes dupliquées par lot)
10. dim_products → alias TOUJOURS pd — jamais dp, d, prod ou tout autre raccourci
    ✓  JOIN marts.dim_products pd ON sd.product_id = pd.product_id → pd.commercial_name · pd.product_category · pd.therapeutic_class
    ✗  JOIN marts.dim_products dp ... → dp.product_category (alias inconnu → erreur SQL)
11. Catégories produits — deux colonnes, deux usages distincts :
    • pd.origin → exactement 2 valeurs : 'Local' · 'Importé'  (NE PAS utiliser 'Sénégal', 'France' ou toute autre valeur inventée)
    ✓  WHERE pd.origin = 'Local'   ✓  WHERE pd.origin = 'Importé'
    ✗  WHERE pd.origin = 'Sénégal' (valeur inexistante — hallucination)
    • pd.product_category  → seulement 2 valeurs (Médicament · Parapharmacie) — utiliser UNIQUEMENT pour comparer médicaments vs parapharmacie
    • pd.therapeutic_class → 14+ valeurs (Antibiotique · Antalgique · Anti-inflammatoire · Antipaludéen…) — utiliser pour "top catégories" / classement / analyses de classes
    ✓  "top 5 catégories" / "quelles catégories génèrent le plus de CA" → GROUP BY pd.therapeutic_class
    ✗  "top 5 catégories" → GROUP BY pd.product_category (retourne toujours 2 lignes maximum)
12. Répartitions / proportion / distribution / breakdown / share → toujours GROUP BY — jamais PIVOT
    Règle absolue : pour toute question de répartition (FR) ou proportion/share/breakdown/distribution (EN)
    → utiliser GROUP BY pour retourner une ligne par catégorie avec la valeur brute
    → JAMAIS SUM(CASE WHEN ...) en colonnes horizontales
    → JAMAIS diviser dans le SQL (le frontend calcule les % automatiquement)
    ✓  SELECT payment_method, COUNT(*) AS nb FROM marts.fct_sales GROUP BY payment_method
    ✓  SELECT pd.is_generic, SUM(sd.total_line_amount_fcfa) AS ca FROM ... GROUP BY pd.is_generic
    ✗  SELECT SUM(CASE WHEN payment_method = 'Espèces' THEN 1 ELSE 0 END) AS especes, ...  (PIVOT interdit)
    ✗  SELECT SUM(...) / SUM(...) * 100 AS part ...  (division entière = 0 en PostgreSQL)
13. sales_rep / agent commercial / sales representative → concept INEXISTANT dans ce modèle de données
    fct_sales n'a pas de colonne sales_rep_id, agent_id, pharmacist_id, employee_id
    Pour segmenter les performances : utiliser client_type, payment_method ou insurer_id
